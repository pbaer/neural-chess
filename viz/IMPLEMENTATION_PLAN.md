## ✅ Resolved decisions (2026-06-06) — these SUPERSEDE the §9 open questions

1. **Featurize for play = FLOAT** (matches the deployed `V3PolicyEngine`, so the tool's moves are identical to the real engine). The training-time **int8 truncation** of planes 17/18/20 is exposed as an *educational toggle* that visibly diffs those planes. Parity test runs both modes.
2. **Single hero model in v1** (no model switcher). The core stays config-driven so a switcher is a purely additive feature later.
3. **Hero = `v3.1-nano`** (pure transformer, **no conv stem**, ~116k params; the conv-stem ablation showed it's smaller *and* stronger — 77% vs sf_easy). Final checkpoint comes from the v3.1 ablation; export the best small v3.1 (`v3.1-eq`, 116k) as the hero.
4. **No parameter editing in v1** (read-only). Noted as a fun future feature — cheap to add since the worker re-runs per call. (MCTS is likewise a future additive seam.)
5. **Desktop/laptop-optimized; mobile explicitly OUT of v1.** BUT — **architectural directive:** the **core must be presentation-agnostic** so a future mobile (or otherwise different) visualization *approach* can reuse it without changes. Do **not** bake desktop assumptions into the core.

## 🧱 Architecture refinement (from decision 5): core ⟂ presentation separation

Split the codebase into a **presentation-agnostic CORE** and one **desktop PRESENTATION** that consumes it. This is a *layering discipline*, not a speculative plugin framework (YAGNI: one presentation now, a clean seam for more):

- **CORE** (knows nothing about rendering, DOM, or desktop layout): the inference **engine + Trace**; the config-driven **model-architecture graph** (nodes/edges/dims/node-IDs); the **content registry** (node-ID → explanation *data*); the **chess/game + play-loop logic** (no UI); and **core state** (model/trace/selection). It exposes a stable contract — `modelGraph`, `forward(board) → Trace`, `content(nodeId)`, `game` API — and nothing else.
- **PRESENTATION** (`presentations/desktop/`, v1): the camera/zoom telescope, rendering (SVG/Canvas/WebGL), visual components, the board UI, layout, and interaction — all *consumers* of the core contract. A future `presentations/mobile/` (or lesson-embed, or a 3D approach) reuses the entire core unchanged.
- *(Content nuance: the explanation **data**/registry lives in core; MDX→React **rendering** lives in the presentation/shared layer.)*

## 🔁 Version-neutrality + the Model Capsule

**Naming rule:** the tool is architecture-version-NEUTRAL. The strings "v3"/"v3.1" appear ONLY in (a) a model's `arch` metadata tag and (b) its `model_id`/folder name (e.g. `v3.1-nano`) — *never* in tool code, types, UI labels, or class names. The current architecture happens to be defined in the repo's `src/v3/model.py`, but the tool never imports it or assumes its shape. (Renames applied: project → "Neural Chess"; `ChessModelV3Engine` → `ModelEngine`.)

**The tool consumes an Model Capsule, never the `.pt`.** A Python exporter (`scripts/export/`, the ONLY thing that imports PyTorch) reads a checkpoint and emits a self-describing, **versioned** Model Capsule. This decouples the tool from PyTorch *and* from any single architecture, and is the seam that absorbs future model/format tweaks.

**Model Capsule = the tool's contract (one per `model_id`):**
- `manifest.json`
  - `capsule_version` (int) — the FORMAT version, *separate* from the model's `arch`. The tool declares a supported range, ignores unknown *optional* fields (forward-tolerant), and fails loudly with a clear message outside the range.
  - `arch` (string, **display-only** — e.g. `"v3.1"`), `model_id`, `param_count`, `created`, weights `sha256`/`bytes`.
  - `graph` — an ORDERED, self-describing architecture description: a list of stages `{ id, kind, dims, weights:[tensorName…], reads:[upstreamId…] }`. `kind` ∈ a known vocabulary (`input_planes`, `stem_conv`, `embed`, `tokenize`, `pos_emb`, `block`, `attn_geo`, `ffn`, `layernorm`, `policy_head`, `value_head`, …). **Connectivity and dims come from here — never hardcoded.**
  - `tensors` — index `[{ name, shape, dtype:'f32', offset, length }]` into `weights.bin`.
- `weights.bin` — concatenated little-endian f32 (BatchNorm folded at export).
- `config.json` — the raw checkpoint config, retained for reference/display.

**How this delivers the flexibility you asked for:**
- The tool drives everything off `kind` via three registries: **`compute[kind]`** (the hand-written op), **`render[kind]`** (the visual panel), **`content[kind]`** (the MDX explanation). The model-graph the UI walks is built *from* `manifest.graph`, not from assumptions.
- A model tweak expressible in existing kinds (different stem, more/fewer blocks, geometry-bias off, new dims, a wider head) → **zero tool changes** — the Model Capsule graph + config drive it.
- A genuinely NEW op kind → a **localized** add (one `compute` fn + one `render` panel + one `content` card + register the kind), not a rewrite. The forward pass stays a `kind`→fn registry, **not** a generic graph interpreter — preserving the "every op is owned and legible" property and avoiding re-inventing ONNX.
- The Model Capsule format itself evolving is gated by `capsule_version`; the exporter and the tool's `capsule reader` move together but the reader validates required fields and tolerates unknown optional ones.

**Net:** the model architecture and the Model Capsule are both expected to keep changing; the tool's only coupling is to the **Model Capsule contract + the `kind` vocabulary**, both versioned and extensible.

# Neural Chess — Implementation Plan (design only)

## 1. Overview & goals

Build a polished, public, static educational site that lets a self-learner (HS-calculus level) **see and play** a tiny v3 square-token chess transformer. Every value — from the architecture diagram down to a single scalar weight and the literal arithmetic of one attention score — is inspectable, with hover/select explanations of the *how* and *why*. The user plays moves against the model and watches the resulting position flow through the network to its policy move + value prediction.

Three product pillars:
- **Honest, owned math.** Inference is a hand-written TypeScript forward pass (no ONNX) that numerically matches PyTorch under a parity test, so every intermediate shown is real.
- **A continuous zoom telescope.** Architecture DAG → component → heatmap → scalar-with-arithmetic, never separate screens.
- **Play *is* the trigger.** Making a move runs one instrumented forward pass that drives both the game and the visualization.

Scope is **one-shot inference only**; PUCT MCTS is designed-for (additive worker/Trace consumer) but not built.

## 2. Tech stack & key architectural decisions

- **React + Vite + TypeScript**, static SPA → `dist/` deployable to GitHub Pages (`base:'/neural-chess/'`) or Netlify (root). **KaTeX** (with MathML) for formulas, **MDX** for content.
- **Render split by scale:** SVG/DOM for board, UI, curves, small matrices (<~256 cells); Canvas2D for medium heatmaps and the L3 numeric grid; **WebGL (regl/twgl)** for the big tensors (the ~700k-scalar whole-model atlas and the full attention tensor). One **`TensorViewport`** unifies them with a hysteresis LOD switch (~28 px/cell) flipping a WebGL textured quad into a virtualized numeric grid.
- **One shared affine `CameraController`** (`scale, tx, ty`) drives SVG (`transform`), Canvas2D (`setTransform`) and WebGL (uniform) in lockstep so the four LODs are pixel-registered; component entry is a shared-element morph (with a cross-fade fallback across heterogeneous layers).
- **State, two tiers:** `uiStore.ts` (**Zustand**) for small hot selector-driven UI state (`modelId, selectedStage/square/head/scalar, zoomLevel, playState, traceId`); `traceStore.ts` (a **non-reactive module singleton**) holding the immutable multi-MB `Trace`, publishing only a monotonic `traceId` into Zustand. Components pull zero-copy `Float32Array` subviews via getters — React reactivity stays O(1) while data is O(MB).
- **Inference in a Web Worker (Comlink).** Weights load once in the worker; `Trace` `ArrayBuffer`s are **transferred** (zero-copy) to the main thread. Two run modes: lightweight (`policy/value/legalMask`) and `fullTrace`. The math is <1 ms; the worker is for ergonomics, transfer, and a clean MCTS seam.
- **Worker is a pure math kernel** (no chess.js). The **main thread owns chess.js**: it featurizes, builds the `legalMask`, decodes, and un-rotates. The worker receives `{planes: Float32Array(21·64), legalMask: Uint8Array(4672)}` and returns `{policyLogits, policyProbs, value, bestLegalIndex, trace?}`. (For future MCTS, a ported legal-move generator can move into the worker without touching panels.)
- **Config-driven everywhere.** Panels, the content registry, the atlas manifest and the Trace schema are all generated from the checkpoint config (`stem_kernel/stem_blocks/d_model/n_heads/n_blocks/value_hidden`) so the same code renders the pure-transformer hero (v3.1) and conv-stem variants.

## 3. Inference engine + weight export + parity test

### Forward pass (`src/engine/model.ts`), config-driven
Confirmed against `src/v3/model.py`. Signature:
```
forward(planes: Float32Array /*21*64*/, opts:{legalMask?:Uint8Array, trace?:TraceRecorder}) -> ForwardResult
```
Pipeline (each op a pure function in `ops.ts`, reductions accumulated in JS `number`/f64):
- **Stem:** `conv2d(x,Wfold,bfold,k,pad=k>>1)`→ReLU, then `stem_blocks` ResidualBlocks (`conv→ReLU→conv→add→ReLU`, BN already folded). For v3.1 (`k=1, stem_blocks=0`) this collapses to a per-square affine+ReLU = a 32×21 `EmbedMatrix`.
- **Tokenize:** `(C,8,8)→(64,C)`, token `s=h*8+w` (PyTorch `flatten(2)` row-major), `+ pos_emb` (64,C).
- **n_blocks pre-norm blocks:** `t += attn(LN(t,ln1)); t += ffn(LN(t,ln2))`; FFN = `Linear(d→ffn_mult·d)→GELU→Linear(→d)`.
- **GeoSelfAttention (`attention.ts`):** split QKV → `(n_heads,64,head_dim)`; `scores[h] = (q@kᵀ)*head_dim^-0.5 + bias[h]`, where `bias[h][i][j]=rel_bias[h][relIdx[i][j]]` and `relIdx[i][j]=((i//8 − j//8)+7)*15 + ((i%8 − j%8)+7)`. **CRITICAL: bias is added AFTER the 1/√head_dim scale** (PyTorch SDPA passes it as additive `attn_mask`). softmaxRow → `@v` → merge heads → `proj`. If `geometry_bias=false`, omit bias and expect no `rel_bias` tensor.
- **Final LayerNorm**, then **policy head** `Linear(C→73)` per token → flatten `flat[s*73+mt]` (4672) → legal-mask (zero illegal) → softmax → argmax-over-legal; **value head** mean-pool 64 tokens → `Linear(C→value_hidden)→ReLU→Linear(→1)→tanh`.
- **`ops.ts` parity rules:** LayerNorm over last dim, `eps=1e-5`; `gelu(x)=0.5x(1+erf(x/√2))` with a minimax/A&S erf (`erf.ts`, abs err <1e-7) — **NOT** the tanh approximation; Linear weights stored PyTorch `(out,in)` row-major.

### Featurize (`featurize.ts`) — confirmed against `src/v2/featurize.py`
Port `_featurize_white_to_move`/`_featurize_black_to_move` over a `BoardState` adapter (implemented in `play/gameStore.ts` via chess.js, exposing piece map, castling rights, ep square, halfmove, fullmove, repetition, and `pieceTypeAt(sq)` — needed by `decode`). Black-to-move applies a **vertical mirror** (`h=7−rank`, file unchanged; castling/ep mirrored). Provide `applyInt8Truncation(planes, mode)`: `'trained'` does `Math.trunc` (mirrors the int8 storage the model trained on — kills plane-17 fraction, 0.5→0 on plane-20, fullmove→{0,1,2}); `'float'` matches `inference.py` (no truncation). **Resolved (load-bearing correction):** un-rotation of output moves is `rotateSquare(sq)=sq^56` = `chess.square_mirror` = **vertical mirror only** (one specialist draft said `63−sq`, which flips both axes — that is wrong; use `sq^56`).

### Export pipeline (`scripts/export/`, run with `.venv/Scripts/python.exe`)
- `export_model.py`: `torch.load(weights_only=False)`, `eval()`; resolve config over `ChessConfigV3` defaults, then **override `stem_kernel` from `input_conv.weight.shape[-1]`** and **count `stem.N.*` keys for `stem_blocks`** (older configs omit both).
- `fold_bn.py`: for each (conv,bn) pair, `s=γ/√(var+1e-5)`, `W'=W·s`, `b'=β−μ·s`; emit folded conv-with-bias, drop bn tensors. Pass-through fp32: `pos_emb, ln*, qkv/proj, ffn.{0,2}, rel_bias, policy_head, value_fc{1,2}`. Exclude `num_batches_tracked`; **`rel_idx` is non-persistent — recompute in `relIndex.ts`**.
- Emit per model: `manifest.json` (`{format_version, arch:'v3', param_count, folded:true, tensors:[{name,shape,dtype:'f32',offset,length}], scale_assertions}`), `config.json`, `weights.bin` (concatenated little-endian f32 in manifest order). `make_catalog.py` writes top-level `public/weights/models.json` (catalog + `sha256` + `bytes`).

### Parity harness — decomposed (the key decision)
`gen_golden.py` registers forward hooks on every submodule, runs `eval()` in fp32 over a fixed FEN set (startpos; black-to-move; en-passant; near-promotion; mixed castling; threefold-rep) and dumps input planes + every intermediate (stem out, per-block ln/scores/probs/ffn, final_ln, policy logits, value) to `golden.{json,bin}`.
- **(a) forward/op parity** (`forward.parity.test.ts`): feed the SAME planes into the TS engine with full trace; compare stage-by-stage. This isolates arithmetic from featurization.
- **(b) featurize parity** (`featurize.parity.test.ts`): TS featurize vs python featurize+int8 cast, in **both** modes, same FENs.
- **Tolerances:** per-op max-abs <1e-4; final logits max-abs <1e-3 **and** cosine >0.99999; value abs <1e-4; **argmax move index identical**. A `ParityBadge` surfaces pass/fail in-app.

## 4. Visualization system

### Four-LOD telescope (one camera)
- **L0 ArchOverview:** SVG DAG board → 21 planes → stem → tokens+pos-emb → ×N blocks (collapsed, one expandable) → policy + value, with animated `DataFlowEdges`. Click a box → morph to L1.
- **L1 Component view:** per-component panels (tensors as labeled heatmaps with axis ticks).
- **L2 heatmap:** zoom one tensor in `TensorViewport` (WebGL textured quad + scale-bar legend).
- **L3 scalar:** LOD switch → virtualized numeric grid; selecting a cell opens **`ScalarInspector`** showing the float, its math name (`score[h,i,j]`, `W_qkv[h,i,j]`), and the **expanded arithmetic** from the Trace (e.g. score `= q_i·k_j/√d + bias[h,Δr,Δf]`; policy logit `= Σ_c t[s,c]·W[mt,c]+b`).

### Component inventory (config-driven)
`InputPlanesGrid` (21 mini boards, moving-side-at-bottom; constant planes tinted+labeled; ep hot-square; rep 0/0.5/1) + `RotationToggle` (animates real↔mirrored frame, annotates int8 truncation); `StemPanel`/`EmbedMatrix` (renders k=1 embed or 3×3 kernels per config); **`FeatureBoards`** (the honesty crux: C channels-across-64-squares ARE 8×8 feature-boards; one token's C-vector is NOT a board; channel↔token-major toggle); `TokenizePanel` + `PosEmbSimilarity`; `TransformerBlockPage` with `LayerNormStrip`, `QKVHeatmaps`, **`AttentionMacroGrid`** (64×64 as an 8×8 macro-grid of 8×8 boards; per-head + average; default to a single guided query-square before revealing all 64), **`GeometryBiasGrids`** (the principle-clean showpiece: `n_heads×15×15` (Δrank,Δfile) tables with **zero-init-vs-learned** toggle/diff and click-a-displacement → highlight same-offset pairs), `ResidualStream`, `FFNPanel` (live GELU curve); `PolicyHeadPanel` (`PolicyTypeGrid` 64×73 in 56-slide/8-knight/9-underprom bands, `LegalMaskOverlay` three stacked states, `MoveArrowsBoard` un-rotated top-k, `MoveTypeDecoderRing`); `ValueHeadPanel` (`MeanPoolViz`, `ValueDial` with live tanh curve, moving-side perspective).

### Rendering tech & the live trace
Big tensors pack into one **R32F atlas** via `AtlasAllocator` (JSON manifest name→tile rect), colormapped in a fragment shader (diverging centered at 0; sequential for probs; colorblind-safe; lockable shared scale + visible legend). The engine writes every intermediate into the `Trace`; **`TraceController`** (play/pause/step/scrub/speed) walks an ordered op list, highlighting the active L0 box, opening its panel, animating the changed tensor input→output with a diff mode. Because all intermediates are retained (~2–3 MB at full granularity for micro, byte-capped with auto-downgrade), any step and any scalar is instantly inspectable. The same `Trace` powers the play loop's "watch it think."

## 5. Pedagogical / contextual-content system

A dotted **node-ID registry** (`content/registry.ts`) is the single join key between the arch graph, the Trace, and **MDX** content. IDs mirror the model (`stem.input_conv`, `tower.block[i].attn.geo_bias`, `head.policy`, `io.planes.16`, …) and are **instantiated from config** (v3.1-nano → conv card auto-switches to "pure per-square embed", stem-block cards suppressed). Every node has a five-slot card — **what / how / why / principle / dims** — each authored at three **Depth levels** (Intuition / Mechanism / Bare-metal) under one React Depth context. Two interaction tiers: transient hover tooltip (1 line + shape) vs sticky click `Inspector` (full card), plus pinnable cards for side-by-side (e.g. zero-init vs learned bias).

A reusable **MathLens** (one MDX per primitive: dot, matmul, softmax, ReLU, GELU, LayerNorm, BatchNorm-note, attention, tanh, mean-pool, linear, residual) renders intuition + KaTeX formula + a **live worked example pulled from the currently selected activation** (e4→d5: literal `q·k` sum → `/√d` → `+geoBias` → softmax row summing to 1.0) + an HS-calculus aside. Chess-specific concept essays cover the 21 planes, the rotation trick (framed as a coordinate change, not chess knowledge), the 8×8×73 move encoding, legal masking ("rules, not strategy"), the geometry-bias showpiece, and the value head. **Guided tour** = an ordered manifest of 9 stops (the seed narrative) that frames the *same* cards, with a "leave tour, explore here" escape. WCAG AA: data-table alternative for heatmaps, keyboard cell navigation, MathML, reading-level-linted prose, `prefers-reduced-motion`.

## 6. Play interface

`react-chessboard` (declarative `position={fen}`, fits React better than imperative chessground) + **chess.js** for rules/SAN/FEN/draws. `gameStore.ts` wraps `Chess` (move/history/fen/turn/humanColor/undo/loadFen/reset) and implements the engine's `BoardState`. `FenInput` (validate via `chess.load`), play either color, free-setup. `usePlayLoop.ts`: legal human move → `gameStore.move()` → on model's turn, main thread featurizes (rotation) + builds `legalMask` → `engine.run(planes, mask, {fullTrace:true})` → returns logits/value/`bestLegalIndex`/Trace → `traceStore.set` + bump `traceId` → `TraceController` animates the flow → decode + un-rotate (`sq^56`) → apply move. `ValueGauge` renders value [−1,+1] (side-to-move) as a tilting eval bar with a plain-language label. **Canonical featurize mode for play = `'float'`** (matches the deployed `V3PolicyEngine`, so the tool's moves equal the real engine), with `'trained'` int8-truncation exposed as an educational toggle that visibly diffs planes 17/18/20 — *(see open question; this overrides the engine draft's "trained-default" in favor of matching live play, but both modes are first-class).*

## Key architectural decisions (summary)

- Inference = a hand-written, config-driven TypeScript forward pass (no ONNX); conv+BatchNorm folded into conv-with-bias at export time so the TS engine has no BN op at all. Parity is gated two ways — (a) identical-planes math parity vs PyTorch fp32-eval golden, (b) separate TS-vs-Python featurize parity — with per-op <1e-4, final-logit cosine >0.99999, and IDENTICAL argmax move index.

- Worker is a pure tensor-math kernel (no chess.js); the main thread owns chess.js and does featurize + legalMask + decode + un-rotation, passing {planes, legalMask} into the worker and getting {logits, value, bestLegalIndex, trace} back via zero-copy transfer. This isolates chess logic, keeps the worker reusable, and leaves a clean additive seam for future PUCT MCTS.

- State is split: Zustand uiStore for small hot UI state, and a NON-reactive module-singleton traceStore holding the immutable multi-MB Trace, published only via a monotonic traceId — never put typed-array activations in reactive/immer state. The single Trace is the source of truth for both the steppable forward-pass animation and the play 'watch it think' loop.

- One shared affine CameraController drives SVG + Canvas2D + WebGL in lockstep for a continuous 4-LOD telescope (arch DAG -> component -> heatmap -> scalar+arithmetic); TensorViewport flips a WebGL R32F-atlas quad into a virtualized numeric grid at ~28 px/cell; the ~700k-scalar whole-model overview and full attention tensor share the GPU atlas path.

- Everything is checkpoint-config-driven: stem_kernel is inferred from input_conv.weight.shape[-1] and stem_blocks from counting stem.N.* keys (older configs omit both); panels, the node-ID content registry, the atlas manifest and the Trace schema are all generated from config so the pure-transformer hero (v3.1) and conv-stem variants render with the same code.

- Content = MDX keyed to a dotted node-ID registry that joins the arch graph, the Trace, and prose; a five-slot taxonomy (what/how/why/principle/dims) at three Depth levels; a reusable MathLens whose worked examples pull the CURRENTLY SELECTED live activations (q.k shown as a literal sum). Geometry bias is the marquee principle-clean exhibit (zero-init vs learned 15x15 per-head tables).

- Resolved a load-bearing conflict: output-move un-rotation for black-to-move is sq^56 (chess.square_mirror = VERTICAL mirror, file unchanged), NOT 63-sq (which flips both axes). The pedagogy draft's 63-sq is incorrect; the engine/featurize code is authoritative.

- Static deploy via Vite (base set for GH Pages, root for Netlify); brotli-precompressed weights fetched lazily per selected model and cached in Cache Storage/IndexedDB keyed by manifest sha256 for clean cache-busting; default model = the tiny hero; GELU is exact-erf, LayerNorm/softmax/conv reductions accumulated in f64.

### 1. Forward pass (`model.ts`, config-driven)
`ModelEngine.forward(planes: Float32Array /*21*64*/, opts: {legalMask?: Uint8Array, trace?: TraceRecorder}): ForwardResult` where `ForwardResult = {policyLogits: Float32Array(4672), policyProbs: Float32Array(4672), value: number, bestMoveIndex: number, trace?: Trace}`. Pipeline, every op a pure function in `ops.ts`:
- **Stem:** `conv2d(x, Wfold, bfold, C, 21, k, pad=k>>1)` then ReLU; read `k`, `C`, `stem_blocks` from config. Each stem ResidualBlock = conv→ReLU→conv→add→ReLU (BN already folded in). For v3.1 (`k=1, stem_blocks=0`) this collapses to a per-square affine + ReLU — render whatever the config says.
- **Tokenize:** reshape `(C,8,8)->(64,C)` with token `s = h*8+w` (PyTorch `flatten(2)` row-major), add `pos_emb` (64,C).
- **n_blocks pre-norm blocks:** `t += attn(layerNorm(t,ln1))`, then `t += ffn(layerNorm(t,ln2))`. FFN = `linear(d->4d)`→`gelu`→`linear(4d->d)`, `ffn_mult` from config.
- **GeoSelfAttention** (`attention.ts`): `qkv = linear(x, Wqkv, bqkv)` → split to q,k,v, reshape `(n_heads,64,head_dim)`; `scores[h] = (q@kᵀ)*scale + bias[h]` where `scale=head_dim**-0.5` and `bias[h][i][j]=rel_bias[h][relIdx[i][j]]`, `relIdx[i][j]=((i//8-j//8)+7)*15+((i%8-j%8)+7)`. CRITICAL: bias is added AFTER scaling (SDPA semantics). Then `softmaxRow`→`@v`→merge heads→`linear(proj)`. If `geometry_bias=false`, omit bias.
- **Final LayerNorm.**
- **Policy head:** `linear(t, Wp, bp)` per token → (64,73) → flatten to `flat[s*73+mt]` (4672). Then `legalMask` (zero illegal) → softmax → argmax. Decode via `moves.ts decodeMove(idx, board)`; if board.turn==black, un-rotate with `rotateSquare(sq)=sq^56` on from/to (preserve promotion).
- **Value head:** mean-pool 64 tokens → `linear(value_fc1)`→ReLU→`linear(value_fc2)`→`tanh`.

`ops.ts` notes for parity: LayerNorm over last dim, `eps=1e-5`, accumulate mean/var in JS `number` (f64). `gelu(x)=0.5*x*(1+erf(x/√2))` using a high-accuracy `erf` (Abramowitz–Stegun 7.1.26 or rational minimax, abs err <1e-7) — NOT the tanh approximation. Linear weights stored `(out,in)` row-major: `y[o]=Σ_i x[i]*W[o*in+i]+b[o]`.

### 2. Featurize (`featurize.ts`)
Port `_featurize_white_to_move`/`_featurize_black_to_move`: 12 piece + 4 castling + ep(16) + halfmove/100 cap1 (17) + fullmove/100 cap2 (18) + side=1 (19) + repetition 0/0.5/1 (20). Black-to-move applies vertical mirror (`h=7-rank`, file unchanged; castling/ep mirrored). Provide `applyInt8Truncation(planes, mode: 'trained'|'float')`: `'trained'` does `Math.trunc` (toward zero) to mirror the int8 storage the model was trained on (kills plane-17 fractional, 0.5→0 on plane-20, fullmove→{0,1,2}); `'float'` matches `inference.py` (no truncation). Default `'trained'` for fidelity; the parity reference must use the same mode (see §4). `rotateSquare(sq)=sq^56`.

### 3. Export pipeline (`export_weights.py`, `fold_bn.py`)
Load `.pt` (`weights_only=False`), `model.eval()`. Resolve config from `data['config']` over `ChessConfigV3` defaults, then OVERRIDE `stem_kernel` from `input_conv.weight.shape[-1]` (config omits it on pre-v3.1 ckpts). **BN fold** (`fold_bn.py`): for each (conv,bn) pair (`input_conv/input_bn`, `stem.i.conv1/bn1`, `conv2/bn2`), `s=γ/√(var+1e-5)`; new `W'=W*s[:,None,None,None]`, new `b'=β-μ*s`; emit folded conv-with-bias, drop bn tensors. Pass-through (already fp32): `pos_emb`, all `ln*`, `qkv/proj`, `ffn.{0,2}`, `rel_bias`, `policy_head`, `value_fc{1,2}`. Write **weights.bin** = concatenation of every tensor as little-endian float32 in manifest order; **manifest.json** = `{format_version:1, arch:'v3', param_count, folded:true, tensors:[{name,shape,dtype:'f32',offset,length}], scale_assertions}`; **config.json** = full resolved config. `weights.ts` fetches manifest+bin, slices `Float32Array` views by offset/length into an `NdArray` map.

### 4. Parity harness (`gen_golden.py`, `parity.test.ts`)
Decompose to isolate featurization from the math: (a) **op/forward parity** — `gen_golden.py` registers forward hooks on every submodule, runs `model.eval()` in fp32 on a fixed FEN set (startpos; a black-to-move pos for rotation; en-passant; near-promotion; mixed castling rights; threefold-rep) and dumps input planes + each intermediate (stem out, per-block ln/attn-scores/attn-probs/ffn out, final_ln, policy logits, value) to `golden.{json,bin}`. `parity.test.ts` feeds the SAME planes into the TS engine with full trace and compares stage-by-stage. (b) **featurize parity** — compare TS `featurize` (mode `'trained'`) against Python `featurize`+int8 cast for the same FENs. **Tolerances:** per-op max-abs-err < 1e-4; final policy logits max-abs < 1e-3 AND cosine > 0.99999; value abs < 1e-4; argmax move index must be IDENTICAL. The golden reference and TS both use `'trained'` truncation so the run is self-consistent; a second golden in `'float'` mode documents the inference.py skew.

### 5. Performance & trace memory
Work is tiny: micro (d=64, n_blocks=10, 64 tokens) ≈ a few M FLOPs/pass — well under 1ms in plain JS; budget <10ms is met with margin, so WebGL is reserved for viz, not compute. **Trace bounding** (`trace.ts`): `TraceRecorder` with granularity levels (`outputs|stage|full`); at `full`, micro stores attention probs (10·8·64·64·4B ≈ 1.3MB) + per-block token tensors (~10·64·64·4B) ≈ 2-3MB total, all `Float32Array` (copied, not referenced). Cap total bytes (e.g. 32MB) and downgrade granularity if exceeded; trace is rebuilt per forward and GC'd, so it is bounded by one pass regardless of game length.

**Key decisions:** Decompose parity into two independent tests: (a) feed IDENTICAL input planes into both PyTorch (fp32, eval) and TS to validate the pure forward-pass math, and (b) separately validate TS featurize against Python featurize. This isolates featurization bugs from arithmetic bugs.; Fold conv+BatchNorm2d into a single conv-with-bias at EXPORT time (s=gamma/sqrt(var+1e-5), W'=W*s, b'=beta-mu*s, BN eps=1e-5). The TS engine then has no BN op at all — fewer ops to reimplement and one fewer parity surface.; Infer stem_kernel from input_conv.weight.shape[-1], NOT from the config dict: existing tiny checkpoints predate the v3.1 ablation and omit stem_kernel from their saved config (it silently defaults to 3). Weight shape is the source of truth; render whatever it implies (k=1,stem_blocks=0 = pure transformer).; Reproduce the int8 storage truncation (planes 17/18/20, Math.trunc toward zero) as the default 'trained' featurize mode because that is the distribution the model actually learned; expose a 'float' mode matching inference.py and flag the train/serve skew explicitly.; Geometry bias is added to attention scores AFTER the 1/sqrt(head_dim) scaling (PyTorch SDPA passes it as attn_mask), and rel_idx is recomputed deterministically in TS rather than exported.; Use exact erf-based GELU (not the tanh approximation) and accumulate LayerNorm/softmax/conv reductions in f64 (JS number) to keep per-op error under 1e-4.; Weights ship as a versioned manifest.json (name/shape/dtype/offset/length) + a single little-endian float32 weights.bin blob, plus config.json; Linear weights kept in PyTorch (out,in) row-major layout, documented in the matmul.; Trace is a per-forward, granularity-leveled, byte-capped structure of copied Float32Arrays (outputs|stage|full), so memory is bounded by ONE forward pass (~2-3MB at full granularity for micro) irrespective of how long the game runs.

**Risks/open:** Train/serve featurization skew is real: training data was stored int8-truncated but src/v3/inference.py featurizes in float32 with no truncation. Need a product decision on which the educational tool presents as canonical (recommend 'trained' for honesty, but it diverges from how the live engine currently plays). Affects only planes 17/18/20.; Training used bf16 autocast; weights are saved fp32 and the parity reference runs fp32 eager. This is the right reference, but it means TS (fp32) may differ slightly from the model's bf16 training-time numerics — fine for a deterministic fp32 forward, worth a note in docs.; Legal-move generation and the board model are NOT in this engine — moves.ts only ports the 4672 encode/decode tables and consumes a legalMask. Whoever owns the board/game UI must supply a legal-move source (chess.js or a ported generator); confirm the BoardState interface and en-passant/promotion edge cases match python-chess so legalMask aligns with decode.; decode_move's queen-vs-underpromotion disambiguation needs board.piece_type_at(from_sq) (a pawn reaching last rank defaults to queen). The TS decode therefore needs piece-at-square info from BoardState, not just the index — confirm this is exposed.; erf approximation choice trades code size vs accuracy; a 1e-7 minimax erf should keep GELU within tolerance, but if final-logit cosine ever dips below threshold, the erf precision is the first suspect.; Checkpoints with geometry_bias=false or stem_blocks=0 must be handled (no rel_bias / no stem.* tensors); the exporter and engine must branch on presence, validated by a config-matrix smoke test across nano/micro/v3.1 ckpts.; Need to confirm num_batches_tracked and any non-persistent buffers (rel_idx) are correctly excluded from export; rel_idx is registered persistent=False so it is absent from state_dict and must be recomputed in TS.

### 3. Rendering tech
- **SVG/DOM**: board, pieces, arrows, panels, popovers, axis labels, legends, the L0 DAG, ValueDial, GELU/tanh curves, small matrices (<~256 cells). Crisp, accessible, hit-testable.
- **Canvas2D**: medium heatmaps (single weight matrix up to a few thousand cells, the 64×73 policy grid, individual 64×64 attention head, the L3 numeric grid).
- **WebGL** (regl/PixiJS): the big stuff — the **whole-model weight atlas (~700k scalars)** packed into one R32F **texture** via an `AtlasAllocator` that gives each named tensor a row-major tile, described by a JSON **atlas manifest** (name → {x,y,w,h}); a fragment shader applies a diverging colormap (centered at 0) and pan/zoom is just the uniform transform → GPU colormaps per pixel, trivially 60 fps. The full attention tensor (8 blocks × 4 heads × 64×64 ≈ 131k) renders the same way.

**`TensorViewport`** unifies it: given typed-array + shape + camera + canvas size, it computes visible cells, chooses WebGL quad below the LOD threshold and virtualized Canvas/DOM numbers above it (only visible cells drawn). Colormaps: diverging (signed weights/acts), sequential (probabilities/attention 0..1), categorical (plane legends); colorblind-safe; lockable shared scale + visible scale-bar. A small **`ParityBadge`** asserts the TS forward matches the PyTorch reference.

### 4. Steppable forward-pass trace
The TS engine runs once per position (batch=1) and writes every intermediate into a **`TraceStore`** (planes, stem out, tokens, per-block ln/qkv/scores/geobias/softmax/·V/proj/residual/ffn, final_ln, policy logits/mask/softmax, value). **`TraceController`** (play/pause/step/scrub/speed) walks an ordered op list; each step highlights the active L0 box, opens its panel, animates the changed tensor flowing from input tile to output tile, with a **diff mode**. Because all intermediates are retained, any step and any scalar's arithmetic is instantly inspectable.

### 5. Single-position flow & play loop
**`FlowLayout`**: L0 DAG pinned top, active component panel center, TraceTimeline bottom, board + inspector right. **`PlayBoard`** + **`MoveList`**: user moves → featurize (rotation) → engine → new TraceStore → the whole viz re-flows; a **"watch it think"** mode auto-plays the trace to the predicted move + ValueDial, then the model replies. Designed extensibly so a later PUCT MCTS becomes an additional TraceStore consumer without touching these panels.

A TensorViewport with a hysteresis LOD switch flips a WebGL textured quad into a virtualized numeric grid only when zoomed past ~28 px/cell.; The ~700k-scalar whole-model overview is a single R32F texture packed by an AtlasAllocator (JSON manifest: tensor name -> tile rect), colormapped in a fragment shader, so pan/zoom is a free GPU transform at 60 fps; the full attention tensor uses the same path.; Every scalar is inspectable: a precomputed TraceStore retains all batch=1 intermediates, and ScalarInspector expands the exact arithmetic (attention score = q_i.k_j/sqrt(d)+geobias; policy logit = sum_c t[s,c]*W[mt,c]+b) using real numbers, not just weights.; The GeometryBiasGrids widget is the principle-clean showpiece: n_heads x 15x15 (delta-rank, delta-file) tables with a zero-init-vs-learned toggle and click-a-displacement -> highlight same-offset square pairs on the 8x8 macro-grid of 8x8 boards.; FeatureBoards enforces the honest framing with a channel-major/token-major toggle: C channels-across-64-squares ARE 8x8 feature-boards; a single token's C-vector is NOT a board.; All component panels read stem_kernel/stem_blocks/d_model/n_heads/n_blocks/value_hidden from the checkpoint config and render whatever they find (pure-transformer hero v3.1-eq vs conv-stem variants), confirmed against real shapes: hero = 32d/4h/8blk, geobias (4,225), policy_head (73,32).; A single TraceStore is the source of truth for the steppable/animated forward-pass AND the play-vs-model 'watch it think' loop, and is designed as a consumer interface so a future PUCT MCTS can plug in without changing panels.

**Risks/open:** WebGL float-texture path: R32F sampling needs the OES_texture_float / EXT_color_buffer_float (WebGL2) capability; need a graceful fallback (encode floats into RGBA8 or pre-quantize) for older mobile GPUs, which slightly complicates the colormap shader.; Numeric parity vs PyTorch for the arithmetic popovers depends on the TS engine matching exact ops (BatchNorm eval-fold, LayerNorm over last dim, EXACT erf-based GELU not the tanh approx, SDPA's internal 1/sqrt(head_dim) scaling, int8 plane truncation). ## [pedagogy] Pedagogical content & interaction design: a three-rung progressive-disclosure explanation system (MDX keyed to arch-graph node IDs), live-numbers math lenses, dual guided-tour/free-explore modes, and chess-specific concept essays — all accessible to a HS-calculus public audience.

## 1. Contextual-explanation system

**Two interaction tiers, one content model.** Hover = a lightweight *tooltip* (≤1 sentence + the node's tensor shape, e.g. "Geometry bias — a learned nudge added to attention scores. (8 heads × 15×15)"). Click/select = the **Inspector**, a docked, pinnable panel showing the node's full card. Selection is sticky (drives the heatmap viewport too); hover is transient and never steals focus. A third tier, *pinned cards*, lets a learner park 2–3 explanations side-by-side for comparison (e.g. zero-init vs learned geometry bias).

**Explanation taxonomy — every node gets the same five slots:** `what` (one plain sentence), `how` (the mechanism/arithmetic), `why` (the design rationale + intuition), `principle` (optional Principle-2 note where load-bearing), `dims` (shape + where it sits in the forward pass). Each slot is authored at up to three **depth levels** (see §5).

**Node ID registry.** A single `content/registry.ts` maps stable dotted IDs to metadata and is the join key between the architecture graph, the activation trace, and the MDX content. IDs mirror the model: `io.planes.16` (EP), `stem.input_conv`, `stem.bn`, `stem.block[k]`, `tower.posemb`, `tower.block[i].ln1`, `tower.block[i].attn.qkv`, `tower.block[i].attn.geo_bias`, `tower.block[i].ffn`, `head.policy`, `head.value`. Reading `stem_kernel/stem_blocks/d_model/n_heads/n_blocks/value_hidden` from the checkpoint config lets the registry instantiate the right nodes (e.g. v3.1-nano: kernel=1, 0 stem blocks → the conv card auto-switches to its "pure per-square embed" variant and stem-block cards are suppressed).

**Authoring/storage.** Content is **MDX** under `content/` (tree below), one file per node, with YAML frontmatter (`id`, `title`, `tags`, `mathOps`, `seeAlso`). Prose is plain markdown; live/interactive bits are a small set of MDX components authors compose: `<What>/<How>/<Why>`, `<Depth level=2>`, `<PrincipleNote>`, `<Dims>`, `<MathLink op="softmax"/>`, `<LiveScalar path="tower.block[3].attn.head[0].score[e4][d5]"/>`, `<WorkedExample op="dot" ref="…"/>`, `<Term id="logit"/>`, `<PlaneSwatch idx="16"/>`, `<TryThis>`. This keeps content writable by a non-engineer and decoupled from rendering. `content/glossary.mdx` defines `<Term>` chips (logit, tensor, parameter, activation, embedding…) that pop a definition inline anywhere.

```
content/
  registry.ts            glossary.mdx
  components/  stem/*.mdx  tower/*.mdx  heads/*.mdx  io/planes/00..20.mdx
  math/        dot-product.mdx matmul.mdx softmax.mdx relu.mdx gelu.mdx
               layernorm.mdx batchnorm.mdx attention.mdx tanh.mdx mean-pool.mdx
  concepts/    rotation-trick.mdx geometry-bias-principle.mdx move-encoding.mdx
               legal-masking.mdx value-head.mdx principle-2.mdx
  tour/        manifest.ts  stops/01-board.mdx … 09-value.mdx
```

## 2. Math-explainer popovers (the "MathLens")

One reusable lens, one MDX per primitive: **dot product, matmul, softmax, ReLU, GELU, LayerNorm, BatchNorm, attention-weighting, tanh** (+ mean-pool, affine/Linear, residual-add). Each lens has four parts: **intuition** (1 line), **formula** (KaTeX with MathML fallback), a **live worked example** pulled from the *currently selected* activation, and an **HS-calculus aside** (smoothness/derivative, why softmax sums to 1, why LayerNorm divides by the std). For elementwise ops a tiny draggable function plot (slider input → highlighted output) makes ReLU's kink, GELU's soft kink, and tanh's saturation tangible.

The live example is the differentiator: selecting one attention cell e4→d5 in head 0 makes `<WorkedExample op="dot">` render the actual q·k as a literal, truncatable sum: `q·k = 0.21·(-0.05) + 0.77·0.34 + … = 1.83`, then `÷√d = 1.83/√16 = 0.46`, then `+ geoBias = 0.46 + 0.12 = 0.58`, then the softmax row that turns 64 such scores into weights summing to 1.0. Numbers come straight from the TS forward pass via the same `path` keys the Inspector uses, so math is never hand-waved.

## 3. Guided tour + free explore

**Tour** = an ordered `manifest.ts` of stops, each a `{nodeId, camera/highlight, mdxStop, tryThis}`. The nine stops follow the seed narrative: board → 21 planes (+rotation) → conv/embed stem ("per-square notebooks") → 64 tokens + pos-emb → "look around" (attention + geometry) → "rethink" (FFN) → repeat ×n → policy head (73→4672→mask→softmax→move) → value dial. Each stop scripts focus + a one-action prompt ("drag this weight to zero and watch the move change"), with Next/Back, a progress rail, and a persistent "Leave tour, explore from here" button that drops the learner into free mode on the current node. **Free explore** = click any graph node or board square; breadcrumb trail; a `/`-command palette ("find geometry bias", "show plane 16"); "Explain this" affordance on every visual element. Tour and free share the exact same cards — the tour just sequences and frames them.

## 4. Chess-specific explanations (concept essays)

Five `concepts/` essays, each beginner→bare-metal: **21 planes** (per-plane `io/planes/NN.mdx` with a `<PlaneSwatch>` mini-board; flags that 17/18/20 truncate to int8). **Rotation trick** — animate a black-to-move board mirroring so "side to move is always at the bottom," framed as a coordinate change, *not* chess knowledge; show the un-rotate (63−sq) on the output. **8×8×73 move encoding** — interactive: click a from-square, scrub the 73 move-types (56 sliding = 8 dir×7 dist, 8 knight, 9 underpromotion), see the highlighted to-square; then **legal masking** as rules-only (illegal logits → −∞ before softmax), explicitly "rules, not strategy." **Geometry bias = the principle-clean showpiece** — toggle zero-init vs learned 15×15 per-head tables rendered as heatmaps; narrate that the model *discovered* which (Δrank,Δfile) offsets matter (e.g. knight L-shapes lighting up) purely from games, with nothing injected. **Value head** — mean-pool → MLP → tanh as a "who's-winning dial" in [−1,+1] from the mover's view.

## 5. Progressive disclosure

A global **Depth** control (Intuition / Mechanism / Bare-metal) plus per-card expanders. Level 1 hides math and shows analogies; Level 2 adds formulas and shapes; Level 3 exposes individual scalars, the literal arithmetic, and parameter editing. Default = Intuition; the tour bumps to Mechanism. Depth is a single React context so every card, lens, and heatmap legend responds consistently.

## 6. Accessibility / readability

Target WCAG AA. Semantic landmarks + roving-tabindex graph nav; every node reachable by keyboard; popovers trap focus and close on Esc. Canvas/WebGL heatmaps get a text alternative ("data table" toggle) and a colorblind-safe diverging palette (blue–white–red) with a numeric legend — never color alone. KaTeX emits MathML for screen readers; formulas carry `aria-label` prose. `prefers-reduced-motion` disables tour/flow animations. Body prose pinned to ~8th-grade reading level (linted in CI), min 16px, user font-scaling, ≥4.5:1 contrast, light/dark themes.

**Key decisions:** Single source of truth for explanations: MDX files keyed to a dotted node-ID registry (content/registry.ts) that also joins the architecture graph and the live activation trace — so prose, math, and numbers all address components by the same key.; Two interaction tiers map to two information densities: transient hover tooltip (1 line + tensor shape) vs sticky click-to-select Inspector panel (full card), plus optional pinned cards for side-by-side comparison.; Uniform five-slot taxonomy for every node — what / how / why / principle / dims — each authored at three depth levels controlled by one global Depth context (Intuition / Mechanism / Bare-metal).; Math is taught via a reusable 'MathLens' (one MDX per primitive) whose worked examples pull the CURRENTLY SELECTED live activations through the TS forward pass, so e.g. q·k is shown as the literal sum of products, never abstractly.; Tour and Free-Explore reuse the identical content cards; the tour is only an ordered manifest of stops (mapped 1:1 to the seed narrative beats) that frames/sequences them, with a 'leave tour and explore here' escape hatch.; Registry is checkpoint-config-driven: it reads stem_kernel/stem_blocks/d_model/n_heads/n_blocks/value_hidden and instantiates/suppresses node cards accordingly (e.g. v3.1-nano kernel=1, 0 stem blocks → conv card swaps to the 'pure per-square embed' variant).; The geometry bias is positioned as the marquee principle-clean exhibit: a zero-init-vs-learned 15×15 per-head heatmap toggle that visually demonstrates the model discovering board geometry from games with nothing injected (directly serving Principle 2).; Chess-specific content is rules-only and explicitly disclaims strategy (legal masking, move encoding, rotation framed as a coordinate change, not chess knowledge), keeping the artifact faithful to the project's experimental framing.; Accessibility is a first-class content concern: canvas heatmaps always have a text/data-table alternative and a colorblind-safe diverging palette with numeric legend; KaTeX emits MathML; prose is reading-level-linted in CI.

**Risks/open:** Live worked examples assume the inference/trace layer exposes named scalar paths (e.g. tower.block[i].attn.head[h].score[from][to]); the exact path schema must be co-designed with the forward-pass-engine owner or the MathLens content can't bind to real numbers.; HS-calculus framing of GELU, LayerNorm variance, and BatchNorm running stats vs train-mode is genuinely subtle — risk of either over-simplifying (misleading) or overshooting the audience; needs a content review pass and possibly a 'good-enough mental model' disclaimer pattern.; BatchNorm at inference uses frozen running mean/var, so its 'live worked example' differs conceptually from a per-batch normalization a learner might expect — must decide how honestly to surface this without confusing beginners (and it may not exist at all in v3.1-nano if stem_blocks=0).; Volume of MDX is large (per-plane ×21, per-primitive ×~12, per-node, per-concept, per-tour-stop) — need an authoring-cost estimate and possibly templated generation for the repetitive plane/block cards to keep it maintainable.; Reading-level linting (e.g. automated grade-level scoring) interacts badly with unavoidable technical terms; the Term-glossary chip system must absorb jargon so the linter isn't fighting necessary vocabulary.; Per-head 15×15 geometry-bias heatmaps and 8×8-macro-grid attention views need an agreed colormap/scaling contract with the visualization owner so legends and the MathLens numbers stay consistent across the app.; Tour scripting that says 'drag this weight and watch the move change' depends on whether parameter-editing/re-inference is in scope for v1 of the tool — needs confirmation, with a read-only fallback narration if editing is deferred.; Decision needed on whether tour stops and the seed-narrative wording are localized/translatable later; if so, the MDX/registry split should reserve i18n keys now rather than retrofit.


## [app-arch] App architecture, play loop, directory layout, static deploy, and a 6-milestone roadmap for the v3 chess-net web explorer

## App architecture (React + Vite + TS)

**Two-tier state.** Split state by update frequency and payload size.

- **`uiStore.ts` (Zustand).** Small, hot, selector-driven UI state: `modelId`, `selectedStage` (planes|stem|token|attn|ffn|policy|value), `selectedSquare`, `selectedHead`, `selectedScalar {tensorId, indices}`, `zoomLevel`, `playState`, `traceId`. Zustand is chosen over Redux (no boilerplate, no provider) and over Context (Context re-renders all consumers; Zustand's `useStore(selector)` re-renders only subscribers to that slice — essential when a hover changes one scalar). Crucially Zustand stores are usable **outside React**, so the worker callback and the play loop can `set()` directly.
- **`traceStore.ts` (non-reactive ref + version counter).** The forward-pass `Trace` is large nested typed-array data (for v3-micro-tau: 21·64 input + per-block 64·64 token activations + 8 heads·64·64 attention scores·10 blocks ≈ tens of MB). Never place this in reactive Zustand state — structural sharing/immer would copy megabytes per frame. Instead hold the immutable `Trace` object in a module-level singleton and publish only a monotonic `traceId: number` into `uiStore`. Components subscribe to `traceId` and pull slices via plain getters (`getTrace().blocks[i].attn[head]`), returning `Float32Array` subarrays (views, zero-copy). This keeps React reactivity O(1) while the data is O(MB).

**Inference engine = Web Worker (Comlink).** The hand-written forward pass (`src/engine`) runs in a dedicated worker. Rationale: (1) keeps the UI thread free during full-trace capture; (2) the worker owns the weights (loaded once) and the `Trace` buffers, which are **transferred** (zero-copy `ArrayBuffer`) back to the main thread; (3) it future-proofs MCTS, which needs hundreds of evals per move and must not block. Comlink wraps the worker as `await engine.load(modelId)` / `await engine.run(fen, {fullTrace: boolean})`. Two run modes: lightweight (`policy`,`value`,`legalMask` only — used for fast play) and `fullTrace` (every intermediate — used when the viz is open). The micro model is <1ms of math, so the worker is for ergonomics/transfer, not raw speed.

## Play interface

- **Board: `react-chessboard` + `chess.js`.** `chess.js` owns rules/legality/SAN/FEN/draw detection; `react-chessboard` renders and handles click-and-drag (`onPieceDrop`, `onSquareClick`, custom arrows/highlights for the model's chosen move and top-k policy). Chessground is more powerful but its imperative API fights React; react-chessboard's declarative `position={fen}` is the better fit for a teaching tool.
- **`gameStore.ts`.** Wraps a `Chess` instance: `move()`, `history()` (SAN), `fen`, `turn`, `humanColor`, `result`, `undo`, `loadFen`, `reset`. FEN setup via `FenInput.tsx` (validate with `chess.load`), play either color, free-setup mode.
- **`usePlayLoop.ts`.** On a legal human move → `gameStore.move()` → if it's the model's turn, call `engine.run(fen, {fullTrace:true})` → engine returns the legal-masked argmax move (un-rotated for black-to-move), `value`, top-k policy, and the `Trace` → `traceStore.set(trace)` + bump `traceId` → `TraceStepper` animates the flow → on completion apply the model's move. So **playing a move IS the forward-pass trigger** — one call drives both the game and the visualization. `ValueGauge.tsx` renders the value head ([-1,1], side-to-move perspective) as a tilting eval bar with a plain-language label.

## Build & deploy (static)

Vite SPA build → static `dist/`. Set `base:'/neural-chess/'` for GitHub Pages project sites (root for Netlify). CI (`gh-pages.yml`): build, `export_model.py` for the 3 tiny models, publish `dist/`.
